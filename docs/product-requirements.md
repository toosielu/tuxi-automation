# 图喜 AI 小红书爆款内容生成系统 V1 产品功能文档

## 1. 产品定位

图喜 AI 是面向“小红书虚拟电商陪跑营”和独立订阅用户的 AI 小红书内容生成工厂。

它不是课程后台，也不是传统 admin 系统，而是一个帮助用户把“产品卖点”快速转成“小红书可发布图片 + 文案”的 AI Studio。

核心目标：

```text
用户输入产品和赛道
-> 系统生成可复制封面图
-> 系统生成帖子配图方案
-> 系统生成真人感小红书文案
-> 系统生成商品承接文案
-> 系统生成私信成交话术
-> 用户复制发布和承接成交
```

最终把小红书虚拟电商项目从“卖资料、卖陪跑”升级为：

```text
AI 爆款内容生成系统 + 陪跑服务 + 订阅收费
```

## 2. 当前已实现功能

### 2.1 AI 内容生成工厂工作台

当前页面已经从传统后台改成现代 AI SaaS 风格：

- 顶部导航
- 中央 AI 输入工作区
- 下方结果展示区
- 无传统侧边栏
- 极简、留白、黑白灰、米白、小红书红点缀

核心输入字段：

| 字段 | 说明 | 当前状态 |
| --- | --- | --- |
| 输入方式 | 文字输入为主，后续扩展图片和链接 | MVP 聚焦文字输入 |
| 原始素材 | 用户输入想法、资料、参考内容 | 已实现 |
| 目标赛道 | AI、商业、教辅、母婴等 | 已实现 |
| 产品名称 | 要推广的虚拟产品 | 已实现 |
| 内容目的 | 引流涨粉、私信咨询、店铺点击、产品成交、信任建立 | 已实现 |
| 目标人群 | 高级选项，默认想做副业的新手 | 已实现 |
| 产品卖点 | 高级选项，默认低成本、可复制、适合新手、自动发货 | 已实现 |
| 当前卡点 | 高级选项，默认不知道发什么和怎么承接成交 | 已实现 |

### 2.2 三种生成模式

| 模式 | 当前能力 | 后续升级 |
| --- | --- | --- |
| 文字生帖子 | 根据文字素材生成封面方案、文案、承接、话术 | 接真实 AI 后提升理解能力 |
| 图片生帖子 | 支持上传图片预览和图片描述 | 后续接图片识别 |
| 链接生帖子 | 支持粘贴链接和补充说明 | 后续做链接解析和参考结构提取 |

### 2.3 结果资产

每次生成输出五类资产：

1. 爆款封面方案
2. 小红书帖子配图方案
3. AI 小红书文案
4. 商品承接文案
5. 私信话术

每类结果都以独立卡片展示，支持复制。封面图和帖子配图支持生成并下载 1:1 PNG 图片。

### 2.4 后端接口

当前核心接口：

```http
POST /api/ai/generate
GET /api/history/list
GET /api/history/detail/{id}
```

`POST /api/ai/generate` 返回：

```json
{
  "id": "",
  "covers": [],
  "posts": [],
  "productCopy": {},
  "dmScripts": [],
  "createdAt": ""
}
```

### 2.5 技术结构

后端使用 Java Spring Boot，按 Spring MVC 风格组织：

```text
controller
service
service/impl
mapper
mapper/memory
domain
```

前端目前是静态页面：


## 3. 当前最大短板

当前系统已经有 AI 产品的界面和生成闭环，但核心生成能力仍然偏 MVP。

现在的后端生成逻辑主要是：

```text
用户输入
+ Java 模板规则
-> 生成封面、文案、承接、话术
```

还不是完整的：

```text
用户输入
+ 爆款结构模板库
+ 真实 AI 大模型
+ 去 AI 味改写规则
+ 赛道语气适配
-> 更像真人写、可直接发布的爆款图文
```

所以接下来开发重点不是继续堆自动发帖、CRM、多账号矩阵，而是先把“生成质量”打穿。

## 4. 下一步开发优先级

### 第一优先级：爆款结构模板库

目标：让系统生成内容时不再泛泛输出，而是先选择一套爆款结构。

V1 先内置这些结构：

| 结构 | 用途 |
| --- | --- |
| 痛点避坑型 | 引流、私信咨询 |
| 清单干货型 | 收藏、涨粉 |
| 结果对比型 | 点击、转化 |
| 真实记录型 | 信任建立 |
| 成交承接型 | 产品成交 |

每个结构包含：

- 封面标题公式
- 开头钩子公式
- 正文展开方式
- CTA 强度
- 适合的内容目的
- 风险提醒

### 第二优先级：真实 AI 生成接口

目标：把当前模板生成升级为真实 AI 生成。

当前实现：

1. 已抽象 `AiContentProvider` 接口
2. 默认保留本地模板生成，避免没有 API Key 时系统不可用
3. 已增加 OpenAI-compatible Provider，可接 OpenAI、DeepSeek、SiliconFlow 等兼容 `/chat/completions` 的模型
4. 模型输出使用结构化 JSON
5. 模型调用失败时自动回退到本地模板 Provider

配置方式：

```text
TUXI_AI_PROVIDER=openai-compatible
TUXI_AI_API_KEY=你的模型APIKey
TUXI_AI_BASE_URL=https://api.openai.com/v1
TUXI_AI_MODEL=gpt-4o-mini
```

如果使用 DeepSeek：

```text
TUXI_AI_BASE_URL=https://api.deepseek.com/v1
TUXI_AI_MODEL=deepseek-chat
```

如果使用 SiliconFlow：

```text
TUXI_AI_BASE_URL=https://api.siliconflow.cn/v1
TUXI_AI_MODEL=Qwen/Qwen2.5-72B-Instruct
```

素材库要求见：

```text
docs/material-library-format.md
```

### 第三优先级：封面模板生成器

目标：让封面不只是“方案”，而是直接可下载的小红书封面图。

V1.1 支持：

- 自动生成 1:1 PNG
- 标题、副标题、标签自动排版
- 干货、避坑、清单、对比、真实记录等模板
- 一键下载

### 第四优先级：Supabase 持久化

目标：让历史记录、生成结果、用户使用次数可保存。

建议数据表：

| 表 | 用途 |
| --- | --- |
| generations | 保存每次生成记录 |
| generation_assets | 保存封面、文案、承接、话术 |
| prompt_templates | 保存爆款结构模板 |
| user_usage | 保存用户生成次数 |

### 第五优先级：订阅化能力

可做成订阅权益：

| 功能 | 订阅价值 |
| --- | --- |
| 每日生成次数限制 | 基础订阅 |
| 爆款结构模板库 | Pro |
| 封面 PNG 生成 | Pro |
| 高级赛道语气 | Pro |
| 批量生成 | 高级版 |
| 图片识别生帖 | 高级版 |
| 链接解析生帖 | 高级版 |

## 5. V1 不做范围

短期不做：

- 自动发小红书
- 自动私信
- 自动爬取小红书数据
- 多账号矩阵
- CRM
- 团队权限
- 支付订阅
- 复杂数据看板

这些功能都可以后置。当前最重要的是让用户打开系统后感觉：

```text
这个工具真的能帮我每天生成能发的小红书图文。
```

## 6. 当前开发路线

按照从高价值到低风险的顺序推进：

1. 补齐爆款结构模板库
2. 把结构模板接入当前生成服务
3. 优化生成结果，让文案包含明确结构和更强真人感
4. 抽象 AI Provider，为真实大模型接入做准备
5. 接真实 AI API
6. 接 Supabase 保存生成记录
7. 升级封面模板下载
8. 再做图片识别和链接解析

## 7. 成功标准

第一阶段成功标准：

- 用户只填核心字段也能生成
- 每次生成 3 个封面方案、3 条文案、商品承接、5 条私信话术
- 文案能明显看出爆款结构
- 文案不出现明显 AI 腔
- 生成结果可以复制、封面可以下载
- 重启前历史记录可查看

第二阶段成功标准：

- 接入真实 AI 后，结果质量明显优于固定模板
- 不同赛道生成结果有差异
- 同一产品多次生成不会高度重复
- 输出 JSON 稳定，不破坏前端展示
