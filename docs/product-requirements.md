# 图喜小红书虚拟自动化成交系统产品需求文档

## 1. 产品定位

图喜小红书虚拟自动化成交系统是一套面向“小红书虚拟电商陪跑营”的 AI 运营工作台。它的目标不是展示课程介绍，而是让用户进入系统后，能围绕一条明确链路持续执行：

```text
选择赛道
-> 生成爆款流量方案
-> 匹配可成交产品
-> 生成小红书帖子
-> 生成店铺承接文案
-> 加入发布计划
-> 手动录入流量数据
-> 系统判断下一轮怎么改
```

V1 的核心价值是“让客户看到流量爆款怎么做”。收益承接是第二优先级，用来证明流量不是空流量，而是能进入店铺、私域和资料包成交链路。

## 2. 最小产品目标

V1 只做一个可演示、可交付、可订阅的最小系统：

- 用户输入赛道、人群、阶段、预算和当前卡点
- 系统输出赛道评分，判断这个方向是否值得测
- 系统生成一组爆款选题，包括标题、封面文案、开头钩子、爆款理由和变现路径
- 系统从产品库中推荐适合承接流量的虚拟产品
- 系统生成 8 条小红书图文帖子草稿，包含引流爆款贴和干货信任贴
- 系统生成店铺承接文案和私域 7 天 SOP
- 用户可以把帖子加入发布排程
- 用户发布后手动录入数据，系统给下一轮优化建议

## 3. 用户需求优先级

### 3.1 第一优先级：流量爆款

客户最想看到的是“我发什么更容易有流量”。因此 V1 的重点不是复杂项目管理，而是围绕小红书内容生产做深：

- 自动拆解目标人群痛点
- 自动生成爆款选题
- 自动生成标题、封面、开头、正文和 CTA
- 给出为什么可能爆的判断依据
- 给出发布节奏和复盘指标

### 3.2 第二优先级：收益承接

有流量之后，用户需要知道怎么把流量接到店铺和私域。V1 要提供基础承接方案：

- 推荐可成交的虚拟产品
- 生成商品标题、详情页卖点和交付说明
- 生成私信引导话术
- 生成私域欢迎语、打标签和 7 天跟进 SOP

### 3.3 暂不追求

V1 不追求真实全自动发帖、自动私信、自动爬取小红书数据、多账号矩阵管理和复杂订阅支付。这些功能容易增加风控和开发成本，不适合作为第一版。

## 4. V1 功能模块

### 4.1 项目输入

用户创建一个内容测试项目，填写：

- 项目名称
- 目标赛道
- 目标人群
- 当前阶段：新手起盘、已有账号、已有店铺
- 每日可投入时间
- 启动预算
- 转化目标
- 当前卡点

### 4.2 AI 诊断与赛道评分

系统根据输入生成赛道评分：

- 需求强度
- 变现适配度
- 执行难度
- 预算适配度
- 合规安全度

输出：

- 综合评分
- 推荐等级
- 赛道总结
- 风险提示
- 下一步建议

### 4.3 爆款流量方案

这是 V1 的核心模块。系统生成一组爆款选题，每个选题包含：

- 选题类型：痛点、反差、清单、结果路径、案例拆解、避坑
- 爆款标题
- 封面文案
- 开头钩子
- 流量评分
- 为什么可能爆
- 对应承接产品或变现路径

判断逻辑优先围绕小红书内容平台的几个关键因素：

- 痛点是否具体
- 标题是否有点击欲望
- 封面是否能一眼看懂
- 内容是否能收藏
- CTA 是否能引导私信或店铺点击
- 是否能承接到虚拟产品

### 4.4 产品库匹配

系统从产品库中匹配适合当前赛道和人群的虚拟产品。

产品字段：

- 产品名称
- 分类
- 适合人群
- 产品类型：资料包、模板、产品库、话术包、工作流
- 定价区间
- 交付方式
- 执行难度
- 风险点
- 卖点列表

V1 先保留内存产品库，后续接 Supabase Postgres。

### 4.5 小红书帖子生成

系统根据爆款选题和推荐产品生成 8 条图文帖子草稿。

每条帖子包含：

- 内容类型：引流爆款、干货信任
- 内容角度
- 爆款标题
- 封面文案
- 图片画面建议
- 正文
- CTA

V1 先生成可复制文案、封面文案和图片画面建议，暂不直接生成图片。文案要避免明显 AI 腔，优先使用真实场景、经验复盘、低压 CTA 和具体人群表达。

### 4.6 店铺与私域承接

系统根据主推产品生成：

- 店铺标题
- 商品副标题
- 详情页卖点
- 自动交付说明
- 合规风险提醒
- 私域 7 天跟进 SOP

目标是让内容流量有明确承接口，而不是只停留在“发帖”。

### 4.7 发布排程

用户可以把生成的帖子加入发布计划。

V1 字段：

- 标题
- 封面文案
- CTA
- 计划发布时间
- 发布状态

真实发布先由用户人工完成。系统负责排程和记录。

### 4.8 手动复盘

用户发布后手动录入：

- 曝光
- 点赞
- 收藏
- 评论
- 私信数
- 店铺点击
- 成交数

系统计算收藏率、私信率、成交率，并输出下一轮优化建议。V1 可以先保留需求和界面占位，逐步实现。

## 5. 商业化价值

系统为陪跑营提高溢价的原因：

- 从“教方法”升级为“给工具和作战台”
- 让新手每天知道该发什么，降低执行阻力
- 把陪跑交付标准化，便于规模化服务更多学员
- 后续可以按月订阅 AI 生成次数、产品库、复盘能力和模板更新

适合订阅化的能力：

- 每月爆款选题额度
- AI 帖子生成额度
- 产品库更新
- 店铺承接模板
- 私域 SOP 模板
- 数据复盘建议

## 6. 技术架构

### 6.1 前端

V1 使用原生 HTML、CSS、JavaScript，保证开发速度和演示稳定性。

### 6.2 后端

后端使用 Java Spring Boot，并按 Spring MVC 结构组织：

- controller
- service
- serviceImpl
- mapper
- domain

### 6.3 数据持久化

目标数据库是 Supabase Postgres。

V1 当前阶段可以先用内存 Mapper 跑通功能和界面；当核心链路稳定后，再把项目、产品、帖子、排程和复盘数据迁移到 Supabase。

## 7. V1 不做

- 真实小红书一键发帖
- 自动登录和验证码处理
- 自动私信
- 小红书数据爬取
- 多账号矩阵管理
- 团队权限
- 支付订阅
- 图片直接生成
- 复杂 CRM

## 8. 当前开发重点

第一阶段先完成“爆款内容工作台”：

1. 后端返回爆款选题列表
2. 前端新增爆款流量方案展示区
3. 生成的帖子可以加入发布排程
4. 店铺和私域承接继续保留
5. 测试覆盖 Java API 和前端数据标准化
